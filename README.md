# 📋 CRM Sample Component for Joomla 6

**CRM Sample** — это компонент для Joomla 6, реализующий простой прототип CRM-системы с управлением стадиями компаний, ограничениями переходов и журналом событий. Компонент готов к использованию, доработкам, расширению и интеграции.

## 🧠 Описание и логика работы

Компонент позволяет управлять компаниями, проходящими через последовательность стадий — от первого контакта до полной активации. **Главное правило:** менеджер не может «перепрыгнуть» стадию, не выполнив обязательное действие.

### 📊 Стадии (последовательность)

| Код           | Название      | Условие входа                                    | Обязательное действие для перехода                        |
|---------------|---------------|--------------------------------------------------|-----------------------------------------------------------|
| `Ice`         | Ice           | –                                                | Попытка контакта                                           |
| `Touched`     | Touched       | Разговор с ЛПР                                   | Разговор с ЛПР + комментарий                               |
| `Aware`       | Aware         | Заполнена форма дискавери                        | Заполнение формы дискавери                                 |
| `Interested`  | Interested    | Есть дата и время демо                           | Планирование демо (дата/время)                             |
| `demo_planned`| demo_planned  | Есть запланированное демо                        | Проведение демо (кнопка/ссылка)                            |
| `Demo_done`   | Demo_done     | Демо проведено (<60 дней)                        | Выставление счёта                                          |
| `Committed`   | Committed     | Счёт выставлен                                   | Оплата получена                                            |
| `Customer`    | Customer      | Оплата получена                                  | Первое удостоверение выдано                                |
| `Activated`   | Activated     | Есть удостоверение                               | –                                                          |

### 📝 События (логируются в истории)

- Попытка контакта (`contact_attempt`) — комментарий
- Разговор с ЛПР (`executive_conversation`) — комментарий
- Заполнение дискавери (`discovery_filled`) — потребности, бюджет, сроки
- Планирование демо (`demo_planned`) — дата/время
- Демо проведено (`demo_done`) — ссылка/комментарий
- Счёт выставлен (`invoice_issued`) — номер счёта
- Оплата получена (`payment_received`) — сумма
- Удостоверение выдано (`certificate_issued`) — номер удостоверения

Все переходы проверяются на соответствие правилам, заданным в таблице `#__crmsample_stages` (поле `allow_skip_to`). Если событие является триггером перехода и условие выполнено, стадия компании обновляется автоматически.

## 🧩 Почему компонент, а не плагин/модуль

Для реализации полноценной CRM с собственными страницами, таблицами и сложной логикой оптимальным выбором является **компонент Joomla**:

- Компонент предоставляет полный MVC-контроль, собственные модели, представления и контроллеры.
- Позволяет создавать страницы в административной панели (список компаний, карточка) без привязки к меню.
- Легко расширяется: можно добавить API, CLI-скрипты, новые виды отчётов.
- Имеет собственную структуру БД, не зависящую от других расширений.
- В отличие от модуля (только вывод) и плагина (только реакции на события), компонент — самодостаточное приложение.

## 🚀 Установка

### Требования
- Joomla 6.x (рпротестировано с 6.0.3)
- PHP 8.4+ (совместим с 8.4)
- MySQL 5.7+ / MariaDB 10.3+

Установка через ZIP-архив
1. Скачайте последний релиз компонента из [репозитория](https://github.com/aleks-jgn/joomla-sample-crm/releases/tag/tag-2) (файл `com_crmsample.zip`).
2. В административной панели Joomla перейдите: **Система → Установка → Расширения**.
3. Перетащите архив в область загрузки или выберите файл через кнопку «Обзор».
4. Нажмите «Загрузить и установить».
5. После успешной установки в меню появится пункт в админ меню **Компоненты → CRM Sample → Companies**.

## 🖥 Использование

### Список компаний
После установки перейдите в **CRM Sample → Companies**. Вы увидите таблицу всех созданных компаний.  
- Для добавления компании нажмите кнопку **Создать** на панели инструментов.  
- Для редактирования кликните на название компании или выберите её и нажмите **Редактировать**.  
- Для удаления отметьте компании и нажмите **Удалить**.

### Карточка компании
Карточка содержит следующие блоки:

1. **Название компании** — редактируемое поле (обязательное для заполнения).
2. **Текущая стадия** — отображается цветом:  
   - 🔵 **Ice** — синий  
   - 🟡 **Остальные** — жёлтый  
   - 🟢 **Activated** — зелёный  
3. **Доступные действия** — кнопки, соответствующие возможным событиям на текущей стадии. Нажатие открывает модальное окно с необходимыми полями.
4. **Инструкция / Скрипт** — текстовая подсказка для менеджера (зависит от стадии).
5. **История событий** — хронология всех действий с датами и подробностями.

### Пример работы
1. Создайте компанию «ООО Пример» (стадия **Ice**).  
2. Нажмите **Попытка контакта**, введите комментарий и сохраните — событие запишется, стадия изменится на **Touched**.  
3. После разговора с ЛПР заполните форму дискавери — стадия станет **Aware**.  
4. И так далее до **Activated**.

Все события и переходы фиксируются в истории, которую можно просмотреть в правой колонке карточки.

## 🏗 Архитектура и модель данных

### Структура файлов компонента (основные каталоги)
com_crmsample/  
├── admin/  
│   ├── forms/                # XML-формы (company.xml)  
│   ├── services/             # DI-контейнер (provider.php)  
│   ├── sql/                  # SQL-файлы установки/удаления  
│   ├── src/                  # Исходный код (MVC)  
│   │   ├── Controller/       # Контроллеры  
│   │   ├── Extension/        # Класс расширения  
│   │   ├── Model/            # Модели  
│   │   ├── Table/            # Классы таблиц  
│   │   └── View/             # Представления  
│   └── tmpl/                 # Шаблоны (HTML + модальные окна)  
├── language/                 # Языковые файлы (en-GB, ru-RU)  
└── site/                     # Минимальная фронтальная часть  

### Модель данных (таблицы)

| Таблица                      | Назначение                                                                 | Индексы                          |
|------------------------------|----------------------------------------------------------------------------|----------------------------------|
| `#__crmsample_companies`     | Основные данные компаний (id, name, current_stage, created, modified)     | `current_stage`, `created`       |
| `#__crmsample_stages`        | Справочник стадий, порядок, разрешённые переходы (JSON в `allow_skip_to`) | `stage_code` (уникальный)         |
| `#__crmsample_events`        | Журнал событий (тип, данные JSON, дата)                                   | `company_id`, `created`, `event_type` |
| `#__crmsample_stage_history` | Аудит переходов между стадиями (from, to, user, created)                  | `company_id`, `created`           |

**Связи:**
- `events.company_id` → `companies.id`
- `stage_history.company_id` → `companies.id`
- `stage_history.user_id` → `users.id` (Joomla)

## ⚖️ Масштабирование до 10 000 компаний/день

При росте нагрузки до ~10 тыс. компаний в день (и соответствующем количестве событий) необходимо применить следующие принципы:

1. **Индексация**  
   - Все внешние ключи и поля, участвующие в WHERE/JOIN, уже проиндексированы.  
   - Для таблицы событий особенно важен индекс `(company_id, created)`.

2. **Партиционирование таблицы событий**  
   - Разбить `#__crmsample_events` по месяцам (по полю `created`). Это ускорит выборки за конкретный период и облегчит архивирование старых записей.

3. **Архивирование**  
   - Старые события (например, старше 1 года) можно перемещать в архивные таблицы или удалять после выгрузки в аналитическую систему.

4. **Кэширование справочника стадий**  
   - Таблица `#__crmsample_stages` редко меняется, её можно кэшировать в памяти (Joomla Cache или внешний Redis).

5. **Минимизация блокировок**  
   - Запись событий и обновление стадии выполнять в коротких транзакциях.  
   - При массовой загрузке данных использовать пакетные вставки.

6. **Логирование**  
   - Использовать асинхронную запись в очередь (например, MySQL с флагом «обработано») для событий, чтобы не блокировать UI.  
   - Для аудита переходов сохранять записи синхронно, но с минимальной задержкой.

7. **Шардинг (при необходимости)**  
   - Если объём превысит возможности одного сервера, можно разделить компании по географическому или иному признаку на разные инстансы.
